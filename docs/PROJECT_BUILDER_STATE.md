# Vision

Documenting the journey of becoming an engineer who builds software that matters.

The long-term goal is not to master a specific technology, but to develop the engineering mindset required to design, build, and maintain software that solves meaningful problems.

This repository is intended to become a place where curious minds can learn not only technologies, but also the thought process behind building scalable, maintainable, and impactful software.

# Engineering Principles Learned

1. One commit should tell one logical story.
2. Capability is not responsibility.
3. Code communicates intent.
4. Architecture is assigning responsibilities.
5. Don't abstract too early.
6. Understand the "why" before the "how".
7. Software models changing state.
8. State and behavior should have separate responsibilities.
9. Minimize scope until there is a reason to expand it.
10. Never debug from assumptions—debug from evidence.
11. Choose data representations based on the problem domain.
12. The problem determines the data type—not the programmer's preference.
13. Data types define behavior, not merely storage.

# Session Highlights

## Biggest Insight

Variables answer:

> What information does the software know?

Data Types answer:

> How should that information behave?

Choosing the correct representation of information is one of the earliest architectural decisions in software engineering.

---

# Lesson of This Session

Software does not merely store information.

It must understand how that information should behave.

Data types were invented to define valid operations, reduce ambiguity, and model reality more accurately.
